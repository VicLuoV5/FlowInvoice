# 修复/升级日志

日期：2026-06-23

## 本次目标

把发票工具从“能跑的 MVP”推进到更接近财务工具需要的可靠性：坏文件不能拖垮整批处理，扫描版 PDF 要能进入 OCR，非发票文件不能进入报销明细，金额/税额必须来自发票结构而不是松散数字猜测。

## 已完成改动

1. 合并排版容错
   - 单个 PDF/图片读取失败时不再让整批任务崩溃。
   - 只要存在有效页面，就继续生成合并 PDF，并在结果消息中列出失败文件。

2. 扫描版 PDF OCR fallback
   - PDF 页面先读取内嵌文本。
   - 如果页面无文本，则按配置倍率渲染为临时 PNG，再交给 RapidOCR 识别。
   - 临时 OCR 图片处理后会自动删除。

3. 非发票文件过滤
   - 行程单、滴滴/高德打车行程单、酒店结账单/账单号类文件不再进入报销明细。
   - 保留航空运输电子客票行程单等真实可报销票据入口，避免一刀切误伤。

4. 金额与税额安全升级
   - 计算金额时保留 raw text 的数字边界，不再把 `36.86 1 36.86` 拼成 `136.86`。
   - 优先识别中文大写金额后面的 `¥xx.xx` 作为最终价税合计。
   - 再用 `不含税金额 + 税额 ≈ 价税合计` 配对，避免从明细行、数量、单价里误取。
   - 无法配对时写入“⚠️ 需手动核对”，并在备注中标记“税额需人工核对”。

5. 发票号码修正
   - 发票号码从保留空格的 raw text 中提取，优先取 16-24 位长号码。
   - 避免把纳税人识别号前 8 位误当发票号码，例如 `91320594`、`91440404`。

6. 桌面端失败处理
   - 后台线程增加兜底异常捕获。
   - 成功显示信息弹窗，失败显示错误弹窗。
   - 即使底层抛出未预期异常，按钮也会恢复可点击状态。

7. 网页端上传安全性
   - 上传文件写入临时目录时增加序号前缀。
   - 避免多个同名文件在临时目录中互相覆盖。

8. PyInstaller 运行目录修复
   - 源码运行时仍使用项目目录。
   - 打包为 one-file exe 后，运行数据目录改为 exe 所在目录，避免落到 PyInstaller 临时解包目录。

9. 环境说明修正
   - 真实验证确认 `rapidocr_onnxruntime>=1.3.0` 当前不支持 Python 3.14。
   - `requirements.txt` 已注明 Python 3.10-3.12。

## 新增回归测试

新增文件：`tests/test_processor_regressions.py`

覆盖行为：

- 增值税票只有总额、无法配对税额时，必须进入人工核对，而不是写出假税额。
- 扫描版 PDF 页面无内嵌文本时，必须渲染页面并走 OCR fallback。
- 合并排版遇到坏文件时，必须跳过坏文件并保存有效页面。
- 07 类住宿增值税专用发票必须输出长发票号、`359.64 / 21.58 / 381.22`。
- 01 类交通电子发票必须保留数字边界，不能把数量拼进金额。
- 02 类含冲减行的交通电子发票必须使用最终中文大写金额后的合计。
- 高德/滴滴行程单必须作为非发票跳过。
- 酒店结账单/账单号图片必须作为非发票跳过。

## 本次事故根因

- 旧算法把 PDF 抽取文本中的空格全部删除，破坏了数字边界。
- 分类规则先看业务词，例如“打车”“酒店”，没有先判断文档是否为正式发票。
- 发票号码兜底过宽，允许从纳税人识别号中截取 8 位数字。
- 真实样本验证只看“生成成功”和数量，没有逐行核对字段正确性，这是上轮漏掉问题的直接原因。

## 剩余风险

- `15.pdf` 当前只抽到“扫描全能王 创建”，仍无法判断是否为发票；需要更清晰原件或人工确认。
- 当前规则主要覆盖这批样本中的电子发票、交通票、住宿票、行程单和结账单。新增票种仍应先加样本回归测试再扩展规则。
